1What it does

Imagine a Marketing app that emits leads in its own format, and an ERP that expects customers in a completely different shape. Normally a developer writes and deploys code to translate between them. The AI Integration Mapper removes that step.

BEFORE

A developer hand-writes a C# handler that reads one event, reshapes the fields, and republishes the other — then ships and maintains it. Every new connection = more code.

AFTER

An AI agent reads both contracts and proposes a mapping. You review and approve it in the dashboard. NimBus then translates every message automatically — audited end-to-end, with no code.

2The big idea: think once, run forever

The magic is a clean split of labour. The AI does the thinking a single time (authoring a reusable translation). The platform does the doing on every message — with no AI call at runtime, so it's cheap, fast, and perfectly repeatable.

3How it works

The agent is only involved while authoring. At runtime, a NimBus component called the Mapping Executor applies the approved translation, checks the result against the target contract, and forwards it on — riding NimBus's normal routing, ordering, recovery, and audit.

Why this is safeAt runtime there is no LLM call — just a sandboxed JSONata transform plus a schema check. Same input always gives the same output, and a bad result is parked for review, never delivered malformed.

4The mapping lifecycle

Every mapping moves through a few clear states. You only ever act on Draft (approve/reject) and occasionally Active (pause).

StateMeaningWhat you do
DraftAuthored by the agent, awaiting review.Approve or Reject it.
ActiveLive — translating every matching message.Nothing (optionally Pause).
PausedSuspended — messages are held, not dropped.Resume when ready.
StaleThe source contract changed; auto-paused for safety.Ask the agent to re-author, then approve.
RejectedYou declined the proposal.Nothing.

5Before you start

  • .NET 10 SDK — to build and run NimBus.
  • Docker Desktop — runs the Azure Service Bus emulator the demo uses (required for the live run).
  • Node.js 22 — to build the management WebApp UI once.
  • (Optional) ANTHROPIC_API_KEY — lets the agent use Claude to author mappings. Without it, a built-in deterministic author is used, so the demo still works end-to-end.
  • You're on the feature branch. This feature lives on worktree-spec-023-ai-integration-mapper. Run everything from that worktree (or merge it to master first).
Where the code isThe mapper is not on master yet — it's on the branch above. All commands below assume your terminal is in the worktree directory .claude\worktrees\spec-023-ai-integration-mapper.

6See it run — the 6-step live demo

This walks an AI-authored mapping from proposal to a live, translated message landing in the ERP — all in the dashboard.

1

Build the UI & launch the demo

One-time UI build, then start everything with the Aspire app host.

# from the worktree root
cd src/NimBus.WebApp/ClientApp; npm install; npm run build
cd ../../..
dotnet run --project samples/CrmErpDemo/CrmErpDemo.AppHost
2

Open the Aspire dashboard & wait for green

The console prints a dashboard URL. Open it and wait until the resources are healthy — especially provisioner (creates the topology, including the Mapping Zone), nimbus-ops (the WebApp), marketing-api, mapping-zone (the executor), and mapping-agent.

3

The agent proposes a mapping

On startup the mapping-agent reads the registered source and target contracts and submits a Draft mapping: marketing.lead.created.v1 → erp.customer.upsert.v1. (It uses Claude if a key is set, otherwise the deterministic author.)

4

Review & approve in the dashboard

Open the WebApp (nimbus-ops) → Mappings in the sidebar (under Manage). You'll see the Draft with its proposed transform, a short rationale, and worked examples (a real sample lead → the customer it produces). If it looks right, click Approve. The state flips to Active. (See the screen below.)

5

Create a Marketing lead

Send a lead to the Marketing app (find its exact URL in the dashboard — typically port 5085).

# PowerShell
$body = '{ "leadId":"L-1001","firstName":"Ada","lastName":"Lovelace","company":"Acme Analytics","email":"ada@acme.io" }'
Invoke-RestMethod -Method Post -Uri "http://localhost:5085/api/leads" -ContentType "application/json" -Body $body
6

Watch it flow — Marketing → mapper → ERP

In the WebApp dashboard you'll see the chain: the lead is translated to erp.customer.upsert.v1 and consumed by ERP/DataPlatform — the whole hop (source → AI-authored mapper → target) in one audit trail. No code ran that you wrote. 🎉

Try the edge casesInvalid output: if a transform produces something the target schema rejects, the message is parked (not delivered) and is recoverable via Resubmit/Skip in the WebApp. Schema drift: change the source contract and the mapping flips to Stale and pauses until re-authored.

9Map your own two systems

The demo uses Marketing→ERP, but the recipe is general. Whether you drive it through the dashboard, the REST API, or an MCP client (e.g. Claude), the steps are the same:

1

Register both contracts

The source and target event types must exist in the schema registry (each is a name + a JSON Schema). Discover what's registered with GET /api/agent/catalog.

2

Ask the agent to map source → target

Point the Mapping Agent at the pair (or call propose_mapping via MCP). It reads both schemas + sample messages and submits a Draft transform with worked examples.

3

Approve it

Review on the Mappings page and click Approve (or POST /api/agent/mappings/{id}/approve).

4

Done — it runs automatically

Every source message is now translated, validated, and delivered. If the source contract later changes, the mapping goes Stale and asks to be re-authored — so you're never silently shipping wrong data.

Driving it from Claude (MCP)Point Claude Desktop / Claude Code at the NimBus.Mcp server (configured against your running WebApp) and use the propose_mapping and list_mappings tools — the same loop, agent-driven.

Troubleshooting

SymptomWhat it means / fix
Mapping stuck in DraftIt's waiting for you — open the Mappings page and Approve it.
Messages not arriving at the targetCheck the mapping is Active (Draft/Paused/Stale don't run). Check the Mappings page for parked items.
Messages are parkedThe transform output failed the target schema. Inspect the parked item in the WebApp; fix the mapping and Resubmit.
Mapping went StaleThe source contract changed. Re-author the mapping (agent) and approve the new Draft.
Approve returns a conflictThe source schema drifted since authoring — the mapping is marked Stale; re-author.
No ANTHROPIC_API_KEYThat's fine — the deterministic author runs and the demo still completes.
Good to knowThe same mapping works across all three storage backends (in-memory, SQL Server, Cosmos DB). At runtime the Executor never calls an LLM, so cost and latency don't scale with traffic — only authoring (once) does.
