Watch an AI agent join the bus

An external AI agent becomes a first-class participant on your event platform — it discovers the topology, invents a new event type at runtime, publishes a validated message, and settles its work, all in the same audit trail as your coded services. No code written or deployed.

1What you'll see

We extend the CrmErpDemo with one new actor: an AI enrichment agent. When someone creates a contact in the CRM, the agent wakes up, asks Claude to classify it (industry, lead score), invents a brand-new event type crm.contact.enriched.v1 the first time it runs, publishes the enriched result, and the ERP/DataPlatform consumes it — exactly like any coded participant, except it's an LLM.

Why it's interestingThe agent never had code written for it. It joins at runtime through a standard REST API (with OpenAPI) and an MCP server — the same way a Claude client would call any tool — and everything it does shows up in the normal dashboard and audit trail.

2What an agent can do

The platform gives an agent six capabilities — the full lifecycle of being on the bus:

1Discoverlist endpoints & event types
2Defineregister a new event type from a JSON Schema — no C#
3Subscribeexpress interest in an event type
4Receivepull the next matching message (long-poll)
5Publishsend a message, auto-validated against its schema
6Settlemark the work done / failed — tracked end-to-end

3The demo flow

4Before you start

  • .NET 10 SDK
  • Docker Desktop — runs the Azure Service Bus emulator the demo uses.
  • Node.js 22 — to build the management WebApp UI once.
  • (Optional) ANTHROPIC_API_KEY — lets the agent use Claude. Without it, a deterministic classifier runs and the demo still completes.
  • On the feature branch — run from the worktree .claude\worktrees\spec-023-ai-integration-mapper (it carries this demo plus a fix that keeps the agent's long-poll from timing out).

5Run the demo

1

(Optional) give the agent a Claude key

Set it on the app host — it forwards the key to the agent. Skip this to use the deterministic classifier.

dotnet user-secrets set "ANTHROPIC_API_KEY" "sk-ant-..." --project samples/CrmErpDemo/CrmErpDemo.AppHost
2

Build the UI & launch the app host

# from the worktree root
cd src/NimBus.WebApp/ClientApp; npm install; npm run build
cd ../../..
dotnet run --project samples/CrmErpDemo/CrmErpDemo.AppHost
3

Open the Aspire dashboard & wait for green

The console prints a dashboard URL. Wait until these are healthy: provisioner (builds the topology incl. the Agent Zone), nimbus-ops (the WebApp), crm-api, erp-api, agent-zone (parks events for the agent), and enrichment-agent.

4

The agent comes online

In the enrichment-agent logs you'll see Subscribed to CrmContactCreated. followed by quiet long-polling — it's waiting for work.

5

Create a CRM contact

Use the CRM web UI (the crm-api resource in the dashboard), or POST one directly. The exact Contact fields are in the crm-api OpenAPI/Swagger — adjust the body to match.

# PowerShell — endpoint: POST /api/contacts on crm-api (find its port in the dashboard)
$body = '{ "firstName":"Ada", "lastName":"Lovelace", "email":"ada@acme.io", "company":"Acme Analytics" }'
Invoke-RestMethod -Method Post -Uri "http://localhost:<crm-api-port>/api/contacts" -ContentType "application/json" -Body $body
6

Watch the agent work

In the enrichment-agent logs you'll see the loop: receive the parked CrmContactCreated → classify (Claude or deterministic) → define crm.contact.enriched.v1 (first run only) → publish the enriched event → settle the original. The ERP/DataPlatform then consumes the enriched event.

7

See it in the dashboard

Open the WebApp (nimbus-ops). The new crm.contact.enriched.v1 event type appears, and the full chain — coded service → AI agent → coded service — is visible in one audit trail with metrics. 🎉

Talking pointNobody wrote or deployed code for the agent. It discovered the platform, defined a new contract at runtime, and participated under the platform's existing ordering, recovery, and audit guarantees — purely through the agent API.

6Drive it yourself from Claude (MCP)

The scripted agent uses the REST API, but you can run the same loop by hand from an MCP client (Claude Desktop / Claude Code) pointed at the NimBus.Mcp server (configured against the running nimbus-ops). Ask Claude to:

discover_topology  →  define_event_type  →  subscribe
   →  (create a CRM contact)  →  receive_messages  →  publish_event  →  settle_message

Everything Claude does lands in the same dashboard and audit trail under its agent id.

Troubleshooting

SymptomFix
Agent log spams TimeoutRejectedException on receiveYou're on an old build. The long-poll window must stay under the 10s HTTP resilience timeout — the branch ships the fix (5s). Pull latest.
First subscribe took ~15sCold start — the WebApp warming its first connection. The standard retry recovers it; it won't recur once warm.
Agent uses the deterministic classifier, not ClaudeNo ANTHROPIC_API_KEY on the app host — set it (Step 1) and restart. The demo still works without it.
Nothing happens after creating a contactCheck agent-zone and enrichment-agent are healthy, and that the contact actually published CrmContactCreated (visible in the dashboard).
Enriched event never reaches ERPConfirm the provisioner finished — it creates the Agent Zone → DataPlatform forward that routes crm.contact.enriched.v1.
