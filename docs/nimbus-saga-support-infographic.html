01 · Answer

What “support” means today

The platform supports the saga pattern through application services; it does not absorb workflow logic into the NimBus framework.

Not built in

Framework-owned saga

NimBus does not create, persist, correlate, migrate, or visualize a business-process state machine on the application’s behalf.

No Saga<TState> No state-machine DSL No saga store No automatic compensation No saga dashboard
Available now

Application orchestrator

A dedicated subscriber owns workflow state and reacts to events using ordinary handlers and domain code.

03 · Boundary

Who owns what?

This separation is the core architectural choice. NimBus guarantees message mechanics; the application remains authoritative for business truth.

NimBus ownsMessaging infrastructure

01
Ordered delivery per workflow session
02
Retries, dead-lettering and failure classification
03
Blocked sessions and deferred FIFO replay
04
Outbox dispatch and scheduled delivery
05
Resolver audit, lineage, metrics and operator recovery

Application ownsBusiness process

01
Workflow state and its database schema
02
Valid transitions and domain invariants
03
Idempotency for state changes and side effects
04
Timeout meaning and deadline policy
05
Compensation and completion semantics
Important distinction: PendingHandoff tracks one message waiting on external work. Request/response waits for one answer. Neither is a multi-step saga state machine.
04 · Gaps

What keeps it from feeling complete

The primitives are credible, but teams still need conventions and reliability plumbing before the pattern is production-friendly.

GAP 01

No consumer inbox

Every transition must defend manually against redelivery, resubmit, and partial side effects.

GAP 02

Publishing needs discipline

Handlers must explicitly preserve session, correlation, lineage, and deterministic message IDs.

GAP 03

Timeout API mismatch

Scheduling is concrete-client-only, and scheduled-message cancellation cannot work through the outbox.

GAP 04

No reference workflow

The architecture decision exists, but the planned guide and executable orchestrator sample do not.

GAP 05

No saga-level view

The WebApp shows message history, not authoritative workflow state, deadline, or current step.

GAP 06

Atomicity is provider-specific

SQL can share state and outbox transactions; Cosmos needs its own transactional outbox pattern.

05 · Value

Why saga-ready support matters

The value is strongest in cross-system processes where partial failure is normal and distributed transactions are unavailable.

↻

Process resilience

Orders, refunds, provisioning, and imports can recover after one participant fails.

≡

Per-entity consistency

Sessions prevent concurrent transitions for the same workflow while preserving scale across workflows.

↶

Explicit recovery

Compensation and timeout behavior become visible domain decisions instead of hidden retry side effects.

◎

Operational clarity

Resolver, deferred replay, and operator actions provide a strong recovery surface for long-running flows.

06 · Fit

How far should NimBus go?

The closer the investment stays to messaging reliability and operations, the better it fits the platform.

Carry application-owned saga workflows

Strong fit

Directly exercises NimBus strengths: sessions, failure recovery, audit, and operator intervention.

Orchestration guide, helpers, and test kit

Strong fit

Reduces repeated mistakes without taking ownership of business state or workflow logic.

Optional lightweight orchestration package

Good fit

Useful if it standardizes correlation, timeouts, telemetry, and testing—without introducing a DSL.

Framework-owned saga state machine

Weak fit

Creates coupling, provider complexity, migration concerns, and long-term framework responsibility.

General workflow or BPM engine

Poor fit

Moves NimBus away from its messaging-platform identity and into a fundamentally different product category.
