# NimBus — Simplification & Performance Remediation Plan

**Created:** 2026-07-05 · **Source:** multi-agent audit (find → adversarial-verify → skeptic).
**Scope:** 97 verified findings (16 high-impact) across 23 projects. 21 candidate findings were refuted on review and are listed at the end.

Full interactive report: `.claire/nimbus-audit.html`.

## How this plan is organised

Findings are grouped into six waves by remediation shape, not by subsystem — each wave is a coherent batch you can land together:

| Wave | Theme | Verifiable with |
|---|---|---|
| 1 | Hot-path correctness & allocation fixes | unit tests (no infra) |
| 2 | Store query efficiency (Cosmos RU / SQL round-trips) | conformance suite + live emulator |
| 3 | Frontend data fetching (N+1, waterfalls, races) | vitest + manual |
| 4 | Dead-code deletion | build + full test run |
| 5 | Duplication consolidation | build + full test run |
| 6 | Remaining backlog (mediums/lows) | per-item |

## Working method

- **One fix per git worktree**, branched off `master`, Conventional-Commit, committed locally (not pushed). Branch naming: `fix/<slug>`.
- **TDD**: add or adjust a failing test first, then implement (per repo convention: MSTest, `#pragma warning disable CA1707, CA2007` at test-file top; frontend uses vitest).
- **Build gate**: Release build of the touched project (`-c Release` surfaces `TreatWarningsAsErrors` + StyleCop/Sonar/Meziantou) **and** the relevant test project green before commit.
- **Constraints honoured**: `[Obsolete]` members stay as back-compat bridges; transport stays Azure Service Bus; Newtonsoft.Json remains the serializer; WebApp is enhanced, not rewritten.

## Wave 1 — detailed (in progress this session)

Each of these is a contained, unit-testable change on the message hot path or a genuine bug. Worktree branch in parentheses.

### 1.1 Outbox poison row halts all publishing — `fix/outbox-poison-per-session`
`src/NimBus.Core/Outbox/OutboxDispatcher.cs:48-53`. `DispatchPendingAsync` breaks the whole batch on the first failed row to preserve ordering, but `GetPendingAsync` returns rows oldest-first with no attempt counter — so one deterministically-failing row (payload deeper than `SafeJsonSettings` MaxDepth=32, or a send to a deleted topic) blocks **every session's** outbound messages permanently. NimBus's ordering unit is the *session* (ADR-001), so the fix is a per-session skip-set: on failure, record the row's `SessionId` and skip only later rows for that session, continuing to dispatch other sessions. Session-less rows fail independently. Update the pinned `DispatchPending_FirstFailure_StopsDispatching` test to the per-session contract and add a test proving a poison row in session A does not block session B. *(Full attempt-count-then-park — needs an `IOutbox` schema column — is deferred to Wave 6.1 so this stays a minimal, safe dispatcher-only change.)*

### 1.2 Message body re-deserialized on every access — `fix/servicebus-messagecontext-memoize`
`src/NimBus.ServiceBus/MessageContext.cs:95,415`. `GetContent()` runs `JsonConvert.DeserializeObject<MessageContent>` over the whole body (and `ServiceBusMessage.Body` copies the byte array) on every `MessageContent` access — 3-4× per message. The context is per-message and single-threaded, so memoize the deserialized `MessageContent` in a field. Test in `MessageContextTests` that two `MessageContent` reads return the same instance / deserialize once.

### 1.3 EventTypeId read via full deserialization in log/retry path — `fix/core-eventtypeid-userproperty`
`src/NimBus.Core/Messages/StrictMessageHandler.cs:401,443,449`. `LogInfo`, `LogError` and `CheckForRetry` read `messageContext.MessageContent?.EventContent?.EventTypeId`, forcing a body deserialization purely to name the event. `IMessageContext.EventTypeId` is user-property-backed and is already the authoritative value on outgoing responses. Switch these three call sites to `messageContext.EventTypeId`. Complements 1.2. Test that retry-policy lookup and log scope use the user-property EventTypeId without touching the body.

### 1.4 Every `Publish(IEvent,…)` serializes twice — `fix/sdk-publisher-double-serialize`
`src/NimBus.SDK/PublisherClient.cs:95-102`. The 3-arg overload serializes the event once to derive `messageId`, then `GetMessageStatic` serializes it again; `GetMessageStatic` already computes the *identical* deterministic messageId when passed `null`. Change the overload body to `await Publish(@event, sessionId, correlationId, null);`. Wire output is byte-identical. Test that the produced `MessageId` is unchanged.

### 1.5 SDK registration bug + misleading fail-fast — `fix/sdk-registration`
`src/NimBus.SDK/Extensions/ServiceCollectionExtensions.cs:189-204,271`. Two fixes in one file (grouped to avoid a same-file merge conflict): **(a)** the three-branch `StrictMessageHandler` construction only passes the resolved `IPermanentFailureClassifier` on the first branch — register a classifier with no pipeline/lifecycle-notifier and it is **silently dropped**. Collapse to a single `new StrictMessageHandler(…, retryPolicyProvider, pipeline, lifecycleNotifier, permanentFailureClassifier)` (the ctor params are nullable). **(b)** `AddNimBusOutboxDispatcher`'s fail-fast tells users to call `AddNimBusPublisher`, which never registers `OutboxDispatcherSender`; correct the message to name the registration that actually provides it. Test that a classifier registered without a pipeline reaches the handler.

### 1.6 Frontend caches a rejected promise forever — `fix/webapp-eventtypes-rejected-promise`
`src/NimBus.WebApp/ClientApp/src/hooks/event-types.ts:24-33`. On a failed request the `.then` cleanup never runs, so the rejected promise stays in `pendingRequests` and every later call returns the same rejection until reload. Move the `pendingRequests.delete(endpointId)` into a `.finally`, cache only on success. Vitest: first call rejects, second call retries (issues a new request) rather than returning the cached rejection.
